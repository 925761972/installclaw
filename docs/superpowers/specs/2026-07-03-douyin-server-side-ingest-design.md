# 抖音链接服务端中转设计

## 目标

仅针对抖音链接模式，增加“服务端下载后上传到 MediaKit 再提交任务”的链路，降低抖音临时链接、防盗链、异步拉源失败导致的 `DownloadFailed`。

本版方案明确采用：

- 解析阶段只做解析，不上传
- 点击“开始擦除”后才执行下载与上传
- 服务端临时文件在请求结束时自动清理

## 范围

- 只处理 `sourceType === "douyin"` 的 URL 模式
- 服务端在任务提交阶段完成抖音视频中转上传
- 前端提交任务时最终改用稳定的 `mediakit://...` 文件 ID

## 非目标

- 本次不改造普通公网直链 `direct` 模式
- 本次不改造本地上传模式
- 本次不做失败后自动重试
- 本次不做服务端长期缓存视频文件
- 本次不做 `mediakit://` 远端对象主动删除

## 核心结论

当前问题不是“解析失败”，而是“抖音外链在火山异步处理阶段不稳定”。因此需要把抖音模式从“提交第三方 URL”改成“提交平台内稳定源”。

## 用户流程

1. 用户粘贴抖音分享链接
2. 服务端继续按现有逻辑解析出真实视频地址、标题和时长
3. `resolve` 接口返回解析成功信息，但此时不上传任何文件
4. 用户点击“开始擦除”
5. 服务端在提交任务时识别到这是抖音来源：
   - 下载解析后的真实视频流到服务器临时文件
   - 上传临时文件到 MediaKit
   - 获取稳定 `mediakit://...` 文件 ID
   - 用该稳定 ID 提交字幕擦除任务
6. 请求结束时删除服务器临时文件

## 技术方案

### 解析阶段只返回元信息

现有 `/api/video/resolve` 保留以下职责：

- 解析 URL
- 提取时长
- 生成预览 token
- 生成 `resolvedToken`

抖音模式在这个阶段不做上传，原因是：

- 避免用户只点“解析”不点“提交”时产生无效上传
- 避免提前消耗带宽和处理时间
- 降低无效中转造成的成本

### 返回结构保持轻量

`/api/video/resolve` 保留已有字段：

- `videoUrl`
- `previewUrl`
- `resolvedToken`
- `durationSeconds`
- `sourceName`
- `sourceType`

其中：

- `videoUrl` 用于预览和后续提交阶段的服务端下载源
- `resolvedToken` 用于提交阶段从服务端缓存中安全取回解析结果

本次不在 `resolve` 返回中增加 `ingestedVideoRef`，因为上传动作已经延后到任务提交阶段。

### 前端提交逻辑保持简单

在 `dashboard-studio.tsx` 中：

- 当 `sourceType === "url"` 且解析结果 `sourceType === "douyin"` 时
- 前端仍提交当前解析结果中的 `videoUrl` 和 `resolvedToken`
- 前端不直接参与下载与上传
- 前端无需新增上传状态结构，只需要展示服务端返回的错误信息

### 服务端任务创建逻辑新增抖音中转分支

`/api/jobs` 增加抖音专用分支：

1. 通过 `resolvedToken` 取回服务端缓存的解析结果
2. 若来源为 `douyin`：
   - 先完成积分校验与预留
   - 再下载远程视频到服务器临时文件
   - 再上传到 MediaKit，得到 `mediakit://...`
   - 再调用现有字幕擦除提交逻辑
3. 若来源不是 `douyin`：
   - 沿用现有逻辑

这样后续擦除任务读到的是稳定源，而不是第三方抖音外链。

## 模块拆分建议

### `lib/video-resolver.ts`

继续负责：

- 解析抖音链接
- 抽取真实视频地址
- 给出 `sourceType/sourceName/duration`

不建议把“下载并上传”逻辑继续堆在这里。

### 新增 `lib/douyin-ingest.ts`

建议新增独立模块，负责：

- 下载远程视频流到服务器临时文件
- 校验内容类型、大小和下载超时
- 申请 MediaKit 上传地址
- 将临时文件上传到 MediaKit
- 返回 `mediakit://...`
- 在 `finally` 中删除服务器临时文件

这样职责更清晰，避免 `video-resolver.ts` 继续膨胀。

### `app/api/video/resolve/route.ts`

负责 orchestration：

- 调 `video-resolver`
- 保持只做解析与返回
- 不在这里触发上传

### `app/api/jobs/route.ts`

负责 orchestration：

- 判断是否是抖音来源
- 调 `douyin-ingest`
- 获取 `mediakit://...`
- 继续复用现有任务提交逻辑

## 失败处理

### 抖音解析成功但提交阶段中转失败

直接返回明确错误：

- `抖音视频中转失败，请重试或手动上传`

原因：

- 若继续提交外链，只会回到当前高概率失败的链路
- 与其让用户等很久再失败，不如在提交阶段尽早失败并给出明确兜底方式

### 中转失败后的产品兜底

本次唯一兜底方案：

- 用户手动上传视频文件

不做：

- 自动重试
- 回退到直接提交抖音 URL

## 兼容性要求

- 本地上传模式必须保持不变
- 普通直链 `direct` 模式必须保持不变
- 预览逻辑必须保持可用
- ETA、任务列表、结果预览/下载逻辑不应受影响
- 解析成功但未提交时，不应产生任何中转上传

## 测试策略

### 单元测试

至少覆盖：

- 抖音模式在解析阶段不触发上传
- 抖音模式在提交阶段触发中转上传
- 抖音模式中转失败时返回预期错误
- 直链模式不触发中转
- 服务端临时文件在成功和失败场景下都被清理

### 集成验证

至少验证：

1. 抖音链接解析成功
2. 解析阶段不产生上传
3. 点击提交后才完成服务端中转上传
4. `/api/jobs` 最终使用 `mediakit://...`
5. 任务成功率相较旧抖音外链方案明显提升
6. 请求结束后临时文件被清理

### 回归验证

重点回归：

- 本地上传可正常处理
- 公网直链可正常处理
- 任务 ETA 正常
- 结果预览和下载正常

## 风险与约束

- 服务端中转会增加一次下载和上传耗时，因此抖音提交阶段会比以前更慢
- 服务器需要具备足够的网络带宽与超时配置，才能稳定完成抖音视频流中转
- 若上传到 MediaKit 的过程失败，需要尽早中断而不是进入任务创建阶段
- `mediakit://` 远端对象不在本次实现中主动删除，仍遵循平台自身生命周期

## 本次实施边界

本次完成后，预期达到：

- 抖音 URL 模式不再把第三方临时链接直接交给火山任务
- 抖音处理成功率明显提升
- 用户失败时能快速得到明确提示和手动上传兜底
- 无效解析不会产生无用上传
- 服务器临时文件不会长期堆积

本次完成后，仍不包含：

- 普通公网直链服务端中转
- 自动重试策略
- 统一 URL 中转开关
- `mediakit://` 远端对象主动删除
