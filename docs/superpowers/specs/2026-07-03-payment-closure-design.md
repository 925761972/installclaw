# 支付闭环设计

## 目标

把当前演示态 `mock` 充值流程升级为可商业化上线的真实支付闭环，满足以下目标：

- 首发前端只展示支付宝支付入口
- 首发实际落地支付宝电脑网站支付
- 代码和配置同时预留支付宝扫码支付
- 代码层预留微信支付 provider 接口，但本次不真正接入微信下单
- 积分到账只认服务端异步回调验签成功，不认前端跳转成功页
- 同一订单无论重复回调多少次，都只能入账一次

## 范围

- 替换当前 `mock` 支付流程
- 新增统一支付抽象层
- 实现支付宝电脑网站支付 provider
- 预留支付宝扫码支付能力位和配置位
- 预留微信支付 provider 接口
- 升级订单表结构以支持真实支付订单状态
- 增加服务端支付回调、验签、幂等入账、账变流水
- 调整套餐页支付交互

## 非目标

- 本次不真正接入微信支付
- 本次不做优惠券、满减、会员订阅、自动续费
- 本次不做发票系统
- 本次不做退款原路退回
- 本次不做多币种和国际支付

## 首发策略

### 用户可见层

- 套餐页首发只展示一个支付入口：支付宝支付
- 用户点击后进入支付宝支付流程
- 页面不展示微信支付入口

### 代码层

- 支付抽象层同时支持 provider 扩展
- 首发实现 `alipay`
- `alipay` 下预留两种支付方式：
  - `alipay_web`
  - `alipay_qr`
- 同时预留 `wechat` provider 接口，但实现先留空或返回未启用状态

### 配置层

- 首发默认跑 `alipay_web`
- 后续如需切到扫码支付，可通过配置切换为 `alipay_qr`
- 无需重写订单模型、账务逻辑和回调处理

## 核心原则

- 积分到账以服务端异步回调为唯一可信信号
- 支付成功页只负责展示状态，不负责加积分
- 渠道回调必须先验签、再校验金额和订单状态、再做入账
- 所有入账逻辑必须幂等
- 订单与积分流水必须可追溯

## 订单模型设计

现有 `orders` 表需要从演示订单升级为支付订单。建议保留已有字段并补充以下字段：

- `provider`
  - 支付渠道
  - 首发使用 `alipay`
  - 后续可扩展 `wechat`
- `payment_method`
  - 支付方式
  - 首发支持 `alipay_web`
  - 预留 `alipay_qr`
- `status`
  - 建议统一状态：
    - `pending`
    - `paying`
    - `paid`
    - `closed`
    - `failed`
- `provider_order_id`
  - 渠道侧下单号或商户传给渠道的订单号
- `provider_trade_no`
  - 渠道支付成功后的交易号
- `paid_at`
  - 支付成功时间
- `closed_at`
  - 订单关闭时间
- `fail_reason`
  - 支付失败或关闭原因
- `notify_payload`
  - 回调原始报文，用于排查
- `idempotency_key`
  - 幂等控制键

## 支付抽象层设计

建议新增统一支付服务层，职责分为三层：

### 1. Payment Provider Interface

统一定义支付 provider 能力，例如：

- 创建支付
- 解析支付返回
- 验签回调
- 提取标准化回调字段

### 2. Provider Implementations

本次包含：

- `alipay-provider`
  - 实现电脑网站支付
  - 预留扫码支付
- `wechat-provider`
  - 本次仅提供接口桩或未启用实现

### 3. Payment Service

统一编排以下行为：

- 创建本地订单
- 调用 provider 下单
- 更新订单状态
- 处理异步回调
- 验签与金额校验
- 幂等入账
- 写积分流水

## 前端支付流程

### 套餐页支付发起

用户点击套餐页的支付宝支付按钮后：

1. 前端请求 `/api/checkout`
2. 服务端创建本地订单
3. 服务端根据配置选择 `alipay_web` 或 `alipay_qr`
4. 如果是 `alipay_web`
   - 返回跳转所需数据
   - 前端跳转支付宝收银台
5. 如果后续切到 `alipay_qr`
   - 返回二维码链接或二维码内容
   - 前端展示二维码弹窗

### 支付完成后的页面体验

- 支付完成后用户会回到站内结果页或充值页
- 页面可展示“正在确认支付结果”
- 页面可轮询订单状态用于展示
- 但前端轮询不负责实际加积分

## 服务端支付流程

### 创建订单

`/api/checkout` 的职责变为：

- 校验当前用户和套餐
- 创建本地订单
- 记录 `provider`、`payment_method`、`amount_cents`、`points`
- 调用对应 provider 的下单逻辑
- 返回标准化支付响应给前端

### 支付方式分流

服务端通过配置决定当前支付宝下单方式：

- `alipay_web`
- `alipay_qr`

前端不需要同时理解多种 provider 细节，只消费统一响应结构。

## 回调与验签设计

### 回调入口

新增支付宝异步回调路由，专门接收支付宝服务端通知。

### 回调处理顺序

严格按照以下顺序处理：

1. 读取回调参数
2. 验签
3. 查找本地订单
4. 校验订单状态是否允许继续处理
5. 校验金额是否一致
6. 校验商户订单号和渠道返回信息
7. 进入事务执行入账
8. 更新订单状态为 `paid`
9. 给用户加积分
10. 写入积分流水
11. 保存必要的回调原文和渠道交易号

### 幂等要求

即使支付宝重复通知：

- 同一订单只能从 `pending/paying` 成功推进一次到 `paid`
- 同一订单只能写入一条有效充值流水
- 同一订单只能给用户增加一次积分

### 不可信输入处理

以下情况必须拒绝入账：

- 验签失败
- 订单不存在
- 订单已关闭
- 金额不匹配
- 回调状态不符合成功条件
- 重复处理导致状态已是 `paid`

## 支付结果与账务一致性

### 可信标准

- 唯一可信标准：服务端异步回调验签成功
- 不可信标准：
  - 前端跳转成功页
  - 用户手动刷新页面
  - 前端 query 参数里的“支付成功”

### 账变记录

积分账变必须记录在 `point_transactions` 中，至少保留：

- 用户 ID
- 流水类型 `recharge`
- 积分数量
- 订单 ID
- 金额
- 备注
- 创建时间

## 前端页面调整

首发套餐页建议改成：

- 按钮文案从“选择此套餐”改为更明确的支付动作
- 支付弹窗不再展示“本地演示支付”
- 首发只展示支付宝支付入口
- 若后续切到二维码方式，可在当前弹窗中直接承载二维码区域

## 环境变量设计

建议补充并明确以下配置：

- `PAYMENT_PROVIDER=alipay`
- `PAYMENT_METHOD=alipay_web | alipay_qr`
- `APP_SECRET`
- `PUBLIC_APP_URL`
- `ALIPAY_APP_ID`
- `ALIPAY_PRIVATE_KEY`
- `ALIPAY_PUBLIC_KEY`
- `ALIPAY_NOTIFY_URL`
- `ALIPAY_RETURN_URL`

同时为未来微信预留：

- `WECHAT_MCH_ID`
- `WECHAT_APP_ID`
- `WECHAT_API_V3_KEY`
- `WECHAT_PRIVATE_KEY`
- `WECHAT_CERT_SERIAL`
- `WECHAT_NOTIFY_URL`

## 测试策略

### 单元测试

覆盖以下逻辑：

- 订单状态流转
- 支付 provider 响应标准化
- 回调验签后的幂等入账逻辑
- 重复回调不重复加积分
- 金额不一致时拒绝入账

### 集成测试

覆盖以下流程：

- 创建订单成功
- 支付宝电脑网站支付下单成功
- 异步回调到账成功
- 用户积分余额增加
- 订单状态变为 `paid`

### 手动验证

上线前至少验证：

- 套餐页发起支付
- 跳转支付宝收银台
- 支付后返回站内页面
- 支付宝异步回调成功
- 用户积分到账
- 重复回调不重复到账

## 风险与约束

- 首发只展示支付宝，微信虽然预留接口，但不能在 UI 上误导用户“可用”
- 如果数据库仍然是本地 SQLite，支付闭环即使写完，也不适合正式商业部署；支付与数据库切换最终需要连续推进
- 如果回调地址、密钥、公钥配置错误，会直接导致支付不到账
- 扫码支付这次只做代码与配置预留，不应在首发默认路径里启用

## 本次实施边界

本次支付闭环实施后，预期达到：

- 首发可用真实支付宝电脑网站支付
- 支付到账依赖服务端回调而非前端页面
- 后续可低成本切换到支付宝扫码支付
- 后续可在不推翻架构的前提下接入微信支付

本次实施后，仍不包含：

- 微信真实支付
- 自动退款
- 发票
- 优惠券
- 会员订阅
