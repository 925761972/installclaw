# 原生另存为下载设计

## 目标

为处理结果视频的 `下载` 按钮增加原生“另存为”能力，让用户可以自行选择保存位置并修改文件名。

## 范围

- 修改结果视频的前端下载流程。
- 优先使用浏览器原生保存窗口。
- 浏览器不支持时自动回退到现有下载方式。

## 非目标

- 不新增服务端下载代理。
- 不新增站内“重命名”弹窗。
- 不调整任务列表或预览弹窗之外的交互。

## 用户流程

1. 用户点击任务列表或预览弹窗中的 `下载`。
2. 前端先将结果视频地址规范化，继续沿用现有默认命名规则，如 `xxx-clean.mp4`。
3. 如果浏览器支持原生文件系统保存接口，则：
   - 拉取视频内容为 `Blob`
   - 调用原生保存窗口
   - 用户自行选择保存位置并确认最终文件名
4. 如果浏览器不支持原生保存接口，则：
   - 自动回退到当前 `<a download>` 下载方式

## 技术方案

### 前端下载主流程

在 `components/dashboard-studio.tsx` 中将当前的 `downloadResult(job)` 改为：

- 先用 `normalizeResultUrl()` 获取稳定下载地址
- 先用 `deriveResultDownloadName()` 得到默认文件名
- 检查 `window.showSaveFilePicker` 是否可用
- 可用时走原生保存流程
- 不可用或原生保存失败时回退为 `<a download>`

### 原生保存流程

原生保存流程建议抽到 `lib/job-result.ts`，保持组件内逻辑简单。流程如下：

1. `fetch()` 结果视频 URL，拿到 `Blob`
2. 基于默认文件名推导建议文件名和扩展名
3. 调用 `showSaveFilePicker()` 打开系统保存窗口
4. 将 `Blob` 写入用户选择的位置
5. 正常关闭文件句柄

### 回退规则

以下场景自动回退到普通下载：

- 浏览器不存在 `showSaveFilePicker`
- 原生保存接口被当前环境禁用
- 原生保存过程中出现能力不兼容

以下场景不回退，直接提示失败更合理：

- 视频地址拉取失败
- 返回内容不可读
- 用户在保存窗口中主动取消

## 交互与错误处理

- 保持现有 `下载` 按钮文案，不新增额外按钮
- 用户主动取消保存时，不提示报错
- 拉取视频失败时，沿用现有错误提示通道，给出简短失败信息

## 测试

- 补充 `lib/job-result` 的小型单元测试，覆盖：
  - 默认下载名推导不变
  - 能正确提取扩展名
- 手动验证：
  - 支持原生保存接口的浏览器可弹出系统保存窗口
  - 不支持的浏览器仍能正常下载

## 风险与约束

- `showSaveFilePicker` 属于较新的浏览器能力，兼容性有限，因此必须保留自动回退。
- 结果视频下载仍依赖浏览器可直接访问火山返回的结果链接；本次不引入服务端中转。
